# 测试反模式

---

## 核心原则

> **"测试代码的真实行为，而非 Mock 的行为。"**

当测试只验证 Mock 的行为而非实际功能时，它们提供的是虚假的信心，同时无法捕获任何真实的 Bug。

---

## 五大反模式

### 反模式 1：测试 Mock 行为

**问题描述**：验证 Mock 是否存在以及是否被调用，而不是测试组件的实际输出。

**错误做法**：测试只检查 Mock 函数是否被调用了某个参数（例如"API 被调用了吗？"），而不验证实际返回的结果。

**正确做法**：测试组件的真实输出（例如"返回的用户名是否正确？"）。如果你只能验证 Mock 的调用，请重新考虑这个测试是否有价值。

---

### 反模式 2：生产代码中混入测试专用方法

**问题描述**：在生产类中添加仅用于测试设置或清理的方法。

**常见表现**：
- 以 `_` 或 `ForTesting` 开头的方法出现在生产代码中
- 生产类中包含 `_resetForTesting()` 等仅供测试调用的方法

**危害**：
- 增加生产代码复杂度
- 模糊了生产代码和测试代码的边界
- 可能在生产环境意外触发测试逻辑

**解决方案**：将清理逻辑移到测试工具函数中。每个测试使用全新实例，而非调用重置方法。

---

### 反模式 3：不理解依赖就过度 Mock

**问题描述**：在不了解副作用的情况下过度使用 Mock，导致测试通过但隐藏了真实问题。

**错误做法**：将所有依赖（库存、支付、物流、通知）全部 Mock 掉，测试通过了但其实什么都没有真正验证。

**正确做法**：
1. 先用真实实现运行测试，了解行为
2. 仅 Mock 真正的外部服务（如支付网关）
3. 保留内部逻辑的真实实现（如库存服务使用测试数据库）
4. 验证真实的状态变化（如库存数量是否减少）

---

### 反模式 4：不完整的 Mock

**问题描述**：Mock 响应缺少下游代码所需的字段。

**错误做法**：Mock 只返回 `{ id: 1, name: 'Test User' }`，但生产代码还会访问 `email`、`permissions`、`settings` 等字段。测试通过了，但生产环境访问这些缺失字段时就会崩溃。

**正确做法**：
- Mock 应镜像完整的真实 API 响应结构
- 使用工厂方法生成完整的 Mock 对象，提供合理的默认值
- 只需覆盖测试场景关心的字段，其余由工厂填充默认值

---

### 反模式 5：测试作为事后补充

**问题描述**：将测试视为可选的后续工作，而非开发的一部分。

**典型场景**：
- 第 1 天：写 500 行代码
- 第 2 天：再写 500 行代码
- 第 3 天："我们需要发布，测试以后再加"
- 第 30 天：生产环境出现严重 Bug
- 第 31 天："为什么我们没有测试？"

**解决方案**：遵循 TDD——测试是实现的一部分，不是文档。没有测试的功能不算"完成"。功能和测试应一起交付。

---

## 检测清单

检查你的测试是否存在这些警告信号：

| 警告信号 | 对应反模式 |
|----------|-----------|
| 只有 Mock 断言，没有测试实际输出 | 测试 Mock 行为 |
| 生产代码中出现 `_` 或 `ForTesting` 方法 | 测试专用方法混入生产 |
| 每个依赖都被 Mock | 过度 Mock |
| Mock 只返回 `{ success: true }` | Mock 不完整 |
| 功能上线数周后才添加测试文件 | 测试作为事后补充 |

## 快速参考

| 反模式 | 症状 | 修复方法 |
|--------|------|----------|
| 测试 Mock 行为 | 只有 Mock 调用断言，没有行为测试 | 断言实际输出 |
| 测试专用方法 | 生产代码中有 `_reset()`、`_setForTest()` | 使用全新实例 |
| 过度 Mock | 单个测试中有 10+ 个 Mock | 先用真实依赖测试 |
| Mock 不完整 | 只返回最少字段的桩数据 | 使用工厂方法，匹配真实响应 |
| 测试事后补充 | 功能上线后才写测试 | 从一开始就 TDD |
